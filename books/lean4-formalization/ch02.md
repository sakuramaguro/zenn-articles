# 第2章：「型」という名の番人 〜型クラスという「免許証」の正体〜

第1章では `Goals accomplished!` を体験しました。そのとき使った `norm_num` や `ring` は、なぜ「効く」のでしょうか。なぜ `rfl` は `a + b = b + a` を証明できないのでしょうか。その答えはすべて「型」にあります。

本章は「完全に理解して暗記するための章」ではありません。第3章のエラー格闘編で詰まったとき、「あの話はここに書いてあった」と戻ってこられるインデックスを作ることが目的です。分からなくて当然ですので、流し読みで第3章へ進んでください。

## 2.0 本章のゴール ──── ゴール逆算型アプローチ

まず本章の終着点を見ていきます。以下のコードが理解・実行できることが本章のゴールです。

```lean
import Mathlib

-- ゴール①：型クラスという「免許証」の確認
-- 数学：ℕ は加法を持つか？ / Lean：[Add ℕ] のインスタンスが存在するか？
#check (inferInstance : Add ℕ)
-- Add.mk : Add ℕ   ← 自然数には足し算の免許証がある

-- ゴール②：#print で型クラスの構造を解剖する
-- 数学：「加法を持つ」とはどういう構造か / Lean：Add クラスの定義を覗く
#print Add
-- class Add (α : Type u) where
--   add : α → α → α   ← 「add 演算が一つある」だけのシンプルな免許証

-- ゴール③：have によるサブゴール分割
-- 数学：「a ≤ b かつ b ≤ c ならば a ≤ c」を段階的に証明する
-- Lean：have でサブゴールを宣言し、linarith で解く
example (a b c : ℕ) (h1 : a ≤ b) (h2 : b ≤ c) : a ≤ c := by
  have step : a ≤ b := h1
  linarith
  -- No goals ✓
```

「なぜ `inferInstance` で免許証が取り出せるのか」「`#print` で何が見えるのか」「`have` でゴールをどう分割するのか」——これらを一つずつ見ていきます。

:::message
🔗 **第3章への伏線**

`#check (inferInstance : Add ℕ)` が成功するのは、Mathlib が `Add ℕ` のインスタンス（免許証）を事前に登録しているからです。逆に、登録されていない型でこれを試みると `failed to synthesize instance` という型クラスエラーが出ます。このエラーの正体と対処法を、第3章で正面から扱います。
:::

## 2.1 型の階層 ──── `Prop` と `Type` は何が違うか

Lean 4 のすべての値には**型（Type）**があります。

```lean
#check 42         -- 42 : ℕ
#check "hello"    -- "hello" : String
#check true       -- true : Bool
```

ここまでは普通のプログラミング言語と同じです。Lean 4 が決定的に異なるのは、**数学的な命題もまた型として扱われる**という点です。

```lean
-- 数学：「1 + 1 = 2 という主張」/ Lean：Prop 型の値
#check (1 + 1 = 2)       -- 1 + 1 = 2 : Prop
#check (∀ n : ℕ, n ≥ 0)  -- ∀ n, n ≥ 0 : Prop
```

`Prop` は「命題の型」です。そして命題 `P : Prop` の**証明**とは、`h : P` という「`P` という型を持つ値」に他なりません。

数学とLeanの対応を整理します。

| 数学の概念 | Lean の表現 | 具体例 |
|---|---|---|
| 自然数の集合 $\mathbb{N}$ | `ℕ`（型） | `42 : ℕ` |
| 実数の集合 $\mathbb{R}$ | `ℝ`（型） | `3.14 : ℝ` |
| 命題 $P$ | `P : Prop` | `1 + 1 = 2 : Prop` |
| 命題 $P$ の証明 | `h : P` | `rfl : 0 = 0` |
| 「$P$ を証明せよ」 | `example : P := by ...` | ゴール = `⊢ P` |

「証明を書く」とは「命題と同じ型を持つ値を構築すること」——この視点が、本章全体を貫く核心です。

:::message
**💡 数学学部生向け：型と集合の違い**

型は集合よりも厳密です。`Nat`（自然数）と `Int`（整数）は別々の型であり、Lean は自動的に変換しません。数学では $\mathbb{N} \subset \mathbb{Z}$ ですが、Lean では `(42 : ℕ)` と `(42 : ℤ)` は異なる型の値です。変換するには `(42 : ℤ)` と明示するか、`Int.ofNat 42` という変換関数を使います。
:::

:::message
**💡 プログラマ向け：Prop は Bool ではない**

`Prop` は `Bool`（`true`/`false`）とは別物です。`Bool` は実行時に計算できる真偽値ですが、`Prop` は「数学的に証明可能かどうか」を表します。`1 + 1 = 2 : Prop` を証明することと、`(1 + 1 == 2 : Bool)` を評価することは、全く異なる操作です。
:::

## 2.2 型クラスという「免許証」 ──── Lean の最も重要な抽象化

型クラスを一言で言うと、「この型はある演算を持っていますよ」という**免許証**です。

たとえば、足し算 `a + b` を書きたいとします。`a, b` がどんな型の値であっても、足し算が定義されていなければ計算できません。Lean 4 は「型 `α` に足し算の免許証（`[Add α]`）がある場合に限り、`a + b` という記法を認める」という仕組みを取っています。

```lean
-- 数学：「型 α の値同士を足し算できる」という性質
-- Lean：Add α というクラスのインスタンスが存在すること
variable (α : Type) [Add α] (a b : α)
#check a + b  -- a + b : α   ← [Add α] が免許証として機能している
```

`[Add α]` という角括弧の記法が「免許証を要求する」シグナルです。

### `#print` で免許証の構造を解剖します

`#print Add` と入力すると、「足し算の免許証」の定義が覗けます。

```lean
#print Add
-- class Add (α : Type u) where
--   add : α → α → α
```

これだけです。「`add` という二項演算が一つ定義されている」という、非常にシンプルな要求です。Lean 4 の型クラスは、このような「最小の約束事」の積み重ねで構成されています。

:::message
**💡 `#print` で構造体を解剖する**

`#print` は型クラスだけでなく、構造体・定義・定理の内部を覗くための万能コマンドです。

```lean
#print Monoid
-- class Monoid (α : Type u) extends MulOneClass α, Mul α where
--   npow : ℕ → α → α   ← n 乗の定義
--   npow_zero : ∀ x, npow 0 x = 1
--   npow_succ : ∀ n x, npow (n + 1) x = npow n x * x
```

数学のモノイド（結合律と単位元を持つ集合）が、Lean では `MulOneClass`（単位元）と `Mul`（乗法）の継承として表現されていることが読み取れます。
:::

:::message
**💡 プログラマ向け：型クラスとインターフェースの違い**

Java や TypeScript の `interface` に似ていますが、型クラスには**自動的なインスタンス解決**があります。`[Add ℕ]` と書くだけで、Lean が Mathlib から自動的に `ℕ` の `Add` インスタンスを探してきます。明示的に実装クラスを渡す必要がない点が、インターフェースとの大きな違いです。この「自動的に」探してくる仕組みが、型クラスを「免許証」たらしめています。
:::

:::message
**🔢 数学学部生向け：型クラスと代数構造の対応**

数学での「群・環・体」という階層は、Lean では型クラスの継承として実装されています。`Field`（体）のインスタンスは `Ring`（環）のインスタンスでもあり、さらに `AddCommGroup`（加法可換群）のインスタンスでもあります。`[Field ℝ]` と書くだけで、実数の加法・乗法・逆元・分配則のすべての性質が自動的に使えるようになります。これが「免許証の階層」という抽象層を一枚挟んだことで、実数の証明でも抽象的な体の証明でも同じコードが使える理由です。
:::

## 2.3 `inferInstance` と `#check` ──── 免許証を手で確認する

型クラスのインスタンスが存在するかどうかを確認するには、`inferInstance` が使えます。

```lean
import Mathlib

-- 数学：ℕ は可換半環（CommSemiring）の性質を持つか？
-- Lean：CommSemiring ℕ のインスタンスがあるか確認する
#check (inferInstance : CommSemiring ℕ)
-- CommSemiring.mk : CommSemiring ℕ   ← 存在する

-- 数学：ℝ は順序体（LinearOrderedField）か？
#check (inferInstance : LinearOrderedField ℝ)
-- LinearOrderedField.mk : LinearOrderedField ℝ   ← 存在する
```

`inferInstance` が成功すれば「その型クラスの免許証が Mathlib に存在する」ことが確認できます。失敗すれば赤いエラーが出ます。

:::message alert
**💀 `failed to synthesize instance` エラーの正体**

これは第3章で最頻出のエラーです。「型 `α` に指定した型クラスの免許証がない」という意味です。対処法は次のどちらかです。

1. `variable (α : Type) [Add α]` のように角括弧で免許証の要求を追加する
2. 具体的な型（`ℕ`、`ℤ`、`ℝ` など）を使う

第3章では、このエラーが実際に登場する場面を実況中継します。
:::

## 2.4 `simp` タクティク ──── 自動整理の魔法と罠

`simp`（simplify）は、定理データベースを使って式を自動的に簡約するタクティクです。

```lean
import Mathlib

-- 数学：0 + n = n（加法の左単位元）
-- Lean：simp が Nat.zero_add を自動適用する
example (n : ℕ) : 0 + n = n := by
  simp
  -- No goals ✓
```

**実況中継：**

```text
-- simp 適用前の InfoView
n : ℕ
⊢ 0 + n = n

-- simp 適用後の InfoView
Goals accomplished!
```

`0 + n` は `Nat.zero_add` という `@[simp]` タグ付きの補題が自動適用されて `n` になります。

:::message alert
**💀 simp は万能ではない：ループと `simp only`**

`simp` が無限ループ（終わらない）を起こすことがあります。複数の補題が互いを書き換えし合う状況で起きます。

対処法は `simp only [補題名]` で使う補題を限定することです。

```lean
-- ループが心配なとき：使う補題を明示する
example (n : ℕ) : 0 + n = n := by
  simp only [Nat.zero_add]
  -- No goals ✓
```

どの補題が効いているか分からないときは `simp?` タクティクを使うと「実際に使われた補題の一覧」が提案されます。その出力を `simp only [...]` に貼り付けることで、安全に書き直せます。
:::

## 2.5 `have` によるサブゴール分割 ──── 証明の設計図を作る

複雑な証明を「小さなステップの積み上げ」で解決するための道具が `have` です。数学での「まず補題を示す。次にその補題を使って…」という論法を、そのまま Lean に翻訳できます。

```lean
import Mathlib

-- 数学：a ≤ b かつ b ≤ c かつ c ≤ d ならば a ≤ d（3段の推移律）
-- Lean：have で中間ステップを明示する
example (a b c d : ℕ) (h1 : a ≤ b) (h2 : b ≤ c) (h3 : c ≤ d) : a ≤ d := by
  -- ステップ①：a ≤ c を先に示す（have で宣言）
  have hac : a ≤ c := by linarith
  -- ステップ②：a ≤ c と c ≤ d から a ≤ d
  linarith
  -- No goals ✓
```

**実況中継：**

```text
-- have hac 宣言直後の InfoView
a b c d : ℕ
h1 : a ≤ b
h2 : b ≤ c
h3 : c ≤ d
hac : a ≤ c      ← have で宣言した補題がコンテキストに追加される
⊢ a ≤ d

-- linarith 適用後
Goals accomplished!
```

`have hac : a ≤ c` を宣言すると、その証明義務が新しいサブゴールとして現れ、証明が完了するとコンテキストに `hac : a ≤ c` として追加されます。

:::message
**💡 have は「証明の設計図」の基本単位**

第5章では `sorry` と `have` を組み合わせた「`sorry` 駆動開発」を学びます。まず大きな証明の論理構造を `have` で骨格化し、各 `have` を `sorry` で仮置きしてから、ステップ一つずつ穴を埋める——これが長い証明を攻略する戦略的なアプローチです。
:::

## 2.X まとめ ──── 第2章の全対応表と練習問題

### 数学とLeanの対応を表にまとめます

| 数学の概念 | Lean の表現 | 備考 |
|---|---|---|
| 命題 $P$ | `P : Prop` | `1 + 1 = 2 : Prop` |
| 命題の証明 | `h : P` | `h : a ≤ b` |
| 型クラス（代数構造） | `class C (α : Type) where ...` | 「免許証」の本体 |
| 型クラスのインスタンス | `instance : C ℕ := ...` | Mathlib が事前登録 |
| インスタンスの確認 | `inferInstance` | `#check (inferInstance : Add ℕ)` |
| 構造体の内部確認 | `#print C` | `#print Add` で定義を覗く |
| 暗黙の型クラス要求 | `[Add α]` | 角括弧＝免許証の要求 |
| `simp` による自動整理 | `simp` / `simp only [...]` | `@[simp]` タグ付き補題を適用 |
| サブゴール宣言 | `have h : P := by ...` | 中間命題をコンテキストに追加 |
| 数値等式の自動証明 | `norm_num` | 具体的な数の計算 |
| 代数的等式の自動証明 | `ring` | 多項式・環の公理 |
| 線形不等式の自動証明 | `linarith` | 仮定を組み合わせる |
| 構造の内部定義を確認 | `#print 定義名` | 型クラス・定理すべてに使える |

**問1（易）：`#check` で型クラスのインスタンスを確認してください。**

```lean
import Mathlib
-- 問：ℤ（整数）は CommRing（可換環）のインスタンスか？
-- #check と #print を使ってインスタンスと構造を確認してください。
#check (inferInstance : CommRing ℤ)
#print CommRing
```

<details>
<summary>💡 ヒントを見る</summary>

`inferInstance` に型クラスを指定すると、Lean が Mathlib から該当のインスタンスを探してきます。成功すればインスタンスの型が InfoView に表示されます。`#print CommRing` で CommRing の定義を解剖してみましょう。

</details>

<details>
<summary>✅ 解答を見る</summary>

```lean
import Mathlib
-- ℤ は CommRing のインスタンスを持つ
#check (inferInstance : CommRing ℤ)
-- Int.instCommRing : CommRing ℤ   ← Mathlib が登録したインスタンス

-- CommRing の構造を解剖する
#print CommRing
-- class CommRing (α : Type u) extends Ring α, CommMonoid α
-- → Ring（環）と CommMonoid（可換モノイド）の両方を継承している
-- No goals ✓
```

`CommRing ℤ` のインスタンスが `Int.instCommRing` という名前で登録されていることがわかります。`extends Ring α` とあるため、`CommRing` のインスタンスがあれば自動的に `Ring` のインスタンスも使えます——免許証の階層構造です。

</details>

<details>
<summary>🔄 別解を見る</summary>

```lean
import Mathlib
-- 複数のインスタンスをまとめて確認する
#check (inferInstance : Ring ℤ)
#check (inferInstance : AddCommGroup ℤ)
#check (inferInstance : OrderedCommRing ℤ)
-- ℤ は順序可換環（大小関係も持つ）のインスタンスも持つ
```

`inferInstance` を複数並べることで、一つの型がどれだけ多くの「免許証」を持っているかを確認できます。

</details>

**問2（中）：`simp` を使って次の等式を証明してください。**

```lean
import Mathlib
-- 数学：n + 0 = n（加法の右単位元）
example (n : ℕ) : n + 0 = n := by
  sorry -- ここを埋める
```

<details>
<summary>💡 ヒントを見る</summary>

`simp` タクティクを試してみましょう。`Nat.add_zero` という補題が `@[simp]` タグを持っているため、`simp` が自動的に適用します。`simp only [Nat.add_zero]` という形でも同じ結果が得られます。

</details>

<details>
<summary>✅ 解答を見る</summary>

```lean
import Mathlib
example (n : ℕ) : n + 0 = n := by
  simp
  -- No goals ✓
```

`simp` を選んだ理由：`n + 0 = n` は `Nat.add_zero` という `@[simp]` タグ付きの補題そのものです。`simp` がこの補題を自動的に発見して適用します。

</details>

<details>
<summary>🔄 別解を見る</summary>

```lean
import Mathlib
-- simp only で使う補題を明示する（ループ防止版）
example (n : ℕ) : n + 0 = n := by
  simp only [Nat.add_zero]
  -- No goals ✓

-- または exact を使って項モードで書く
example (n : ℕ) : n + 0 = n :=
  Nat.add_zero n
```

`simp only [Nat.add_zero]` は使う補題を明示するため、ループの心配がありません。`exact Nat.add_zero n` はタクティクモードを使わない「項モード（term-mode）」の証明で、より簡潔です。

</details>

**問3（難）：`have` を使ってサブゴールに分割し、証明してください。**

```lean
import Mathlib
-- 数学：a ≤ b かつ b < c ならば a < c（≤ と < の混合推移律）
example (a b c : ℕ) (h1 : a ≤ b) (h2 : b < c) : a < c := by
  sorry -- have を使って証明してください
```

<details>
<summary>💡 ヒントを見る</summary>

`have` で中間命題を宣言してから `linarith` で解く方法を試してみましょう。コンテキストに `h1 : a ≤ b` と `h2 : b < c` があれば、`linarith` が線形算術として `a < c` を自動的に導きます。あるいは `lt_of_le_of_lt h1 h2` という Mathlib の補題を `exact` で使う方法もあります。

</details>

<details>
<summary>✅ 解答を見る</summary>

```lean
import Mathlib
example (a b c : ℕ) (h1 : a ≤ b) (h2 : b < c) : a < c := by
  -- have で「中間命題」の証明義務を明示する
  have key : a ≤ b := h1
  -- key : a ≤ b と h2 : b < c から linarith が a < c を導く
  linarith
  -- No goals ✓
```

`linarith` を選んだ理由：コンテキストに `h1 : a ≤ b` と `h2 : b < c` があれば、`linarith` が線形算術として `a < c` を自動的に導きます。`have key` は冗長ですが、論理の流れを段階的に確認する練習として有効です。

</details>

<details>
<summary>🔄 別解を見る</summary>

```lean
import Mathlib
-- Mathlib の lt_of_le_of_lt を直接 exact で使う
example (a b c : ℕ) (h1 : a ≤ b) (h2 : b < c) : a < c :=
  lt_of_le_of_lt h1 h2

-- または calc ブロックで段階的に書く
example (a b c : ℕ) (h1 : a ≤ b) (h2 : b < c) : a < c :=
  calc a ≤ b := h1
       _ < c := h2
```

`lt_of_le_of_lt` は「`≤` と `<` の推移律」を直接述べる Mathlib の補題です。`calc` ブロックは「$a \leq b < c$」という数学的な式変形をそのまま Lean に翻訳する読みやすい記法で、第5章で多用します。

</details>

## 2.X つまずきポイントQ&A

:::message
**📝 第2章 つまずきポイントQ&A**

**Q1. `rfl` を使ったら `not definitionally equal` と言われた**
→ `rfl` は「両辺が定義を展開すると文字通り同じ式になる」場合にのみ使えます。`a + b = b + a` のような交換法則は定義的な等値ではなく、`ring` タクティクが必要です。エラーメッセージに `definitionally equal` という語が見えたら、`ring` か `simp` か `linarith` を試してください。

**Q2. `by` の中でインデントがずれてエラーが出た**
→ Lean 4 はインデントに厳格です。`by` の後に続くタクティクはすべて同じ深さのインデントでなければなりません。VS Code でタブとスペースが混在していないか確認してください。タクティクが同じ行に続いていることが原因のこともあります。

**Q3. `failed to synthesize instance` エラーが出た**
→ 型クラスの「免許証」がない状態です。`variable (α : Type)` だけ宣言して `a + b` を書こうとしても、`α` に `[Add α]` という免許証がないためエラーになります。`variable (α : Type) [Add α]` と角括弧で要求を追加してください。

**Q4. `Type*` と `Type u` の違いが分からない**
→ `Type u` は宇宙レベル `u` の型の型です。`Type*` は「適切な宇宙レベルを Lean に自動決定させる」という省略記法で、ほとんどの場合 `Type*` を使えば十分です。第4巻・第5章に進むまでは `Type*` と覚えておいて問題ありません。

**Q5. `simp` でループが起きてタイムアウトした**
→ `simp only [...]` で使う補題を明示的に限定してください。どの補題が効いているかわからない場合は、`simp?` タクティクを使うと「実際に使われた補題の一覧」が提案されます。その出力をそのまま `simp only [...]` に貼り付けることで、ループを防ぎながら同じ結果が得られます。
:::

## 著者より

「型クラス＝免許証」という比喩は、Lean 4 の設計思想を最初に理解しようとした時に自然と浮かんだものです。`[Add α]` と書いた瞬間、「この型は足し算の試験を通過している」という感覚が伴うようになり、エラーメッセージが読めるようになってきました。

本章で最も伝えたかったのは、「型クラスは暗記するものではなく、`#print` で覗くものだ」ということです。第3章では、この免許証の仕組みを知っているかどうかが、エラーとの格闘で決定的な差を生みます。

次の第3章では、いよいよ本格的なエラーと向き合います。「なぜ `ring` が使えないのか」「Claude をメンターとしてどう質問するか」——第2章で作ったインデックスが、そこで生きてきます。
